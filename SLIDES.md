## P4 · 最简模型设计

课本将 I/O 软件层分为四层，我们额外加入硬件层使流程更完整：

```
L0  用户层 I/O 软件    sys_read(fd, buf, len) — 发起系统调用
      │  IRP ↓
L1  设备无关软件 (VFS)  路径解析 → 权限校验 → 缓冲分配
      │  IRP ↓
L2  设备驱动程序         构造 IRP → 写控制寄存器 → 进程阻塞
      │  IRP ↓
L4  硬件层 (磁盘控制器)  磁盘寻道 → 扇区读取 → DMA 传输完成
      │  IRQ ↑
L3  中断处理程序 (ISR)   ISR 响应 → 数据拷贝 → copy_to_user → 唤醒进程
      │  DATA ↑
L0  用户层 (数据返回)     read() 返回，用户进程继续执行
```

[备注：这是最简模型，还没有双缓冲、没有 ACL、没有故障注入。只是一个正常的 I/O 流转。]

## P7 · 为什么加入双缓冲

**我们最初考虑了通道（I/O Channel）方案。**

课本中 I/O 控制方式的演进路线：

```
程序直接控制 (轮询) → 中断驱动 I/O → DMA 控制器 → I/O 通道处理器
```

通道的优点：专用 I/O 处理器，有自己的指令集（CCW），能管理多台设备并发 I/O。

通道的代价：CCW 取指/译码/执行循环、CAW/CSW 握手协议、多设备调度仲裁 —— 对教学场景而言过于复杂，且可视化难度大。

**通道的核心思想是什么？** 让 I/O 操作与 CPU 计算并行执行，减少 CPU 干预。

**双缓冲（乒乓缓冲）的核心思想是什么？** 两块内核缓冲区交替读写 —— DMA 往缓冲 A 写数据时，CPU 可以同时从缓冲 B 读取上一轮的结果。同样是「让 I/O 与计算并行」。

| | 通道 | 双缓冲 |
|------|------|------|
| 并行的层面 | 通道程序独立于 CPU 执行，管理多设备 | 两块缓冲区交替，DMA 写一块时 CPU 读另一块 |
| 实现复杂度 | 高（CCW 状态机、设备仲裁） | 中（状态机直观建模） |
| 可视化效果 | 难以在浏览器中呈现 | 缓冲 A/B 颜色高亮 + 粒子动画一目了然 |

**结论：参照通道「减少 CPU 干预、并行传输」的设计思想，结合教学场景的可视化需求，我们选择了双缓冲方案。**

## P8 · 双缓冲的实现

**核心算法 —— 乒乓切换：**

```
初始化: ActiveWriteBuffer = 1, ActiveReadBuffer = 1
TotalChunks = ceil(BytesToRead / 4096)   // 4KB 每块

每次 ISR 响应后:
  if 还有更多数据块:
      CurrentChunk++
      ActiveReadBuffer  = ActiveWriteBuffer     // 刚写入的变为读取源
      ActiveWriteBuffer = 3 - ActiveWriteBuffer // 1 ↔ 2 翻转
      编程下次 DMA → 硬件开始写新的 ActiveWriteBuffer
      进程 BLOCKED → 回到 Driver 层继续循环
  else:
      进程 RUNNING，read() 返回
```

**前端粒子动画**：乒乓切换发生时，Canvas 生成 5 个粒子沿二次贝塞尔曲线从硬件层卡片飞向目标缓冲区卡片，直观展示「数据正在流向哪个缓冲区」。

## P9 · 为什么加入 ACL 权限校验

课本 VFS 层的核心职责之一就是**权限检查**。我们在设计完基本流转后，加入了完整的文件系统安全校验。

**四级校验管线：**

```
路径穿越检测 → 文件存在检查 → 敏感文件保护 → Unix 权限位检查
```

**支持多身份：**

| 身份 | uid | 特权 |
|------|-----|------|
| Root 超级用户 | 0 | 绕过所有权限检查 |
| 普通用户 user1 | 1000 | 受 owner/group/other 权限位约束 |

**路径穿越攻击防御：** 解析 `../../etc/shadow` 中的 `..` 段为规范绝对路径 `/etc/shadow`，判定逃逸出 `/home/user1/` 主目录范围，返回 EFAULT。

## P10 · ACL 的实现细节

**模拟文件系统数据库（FileSystemDB）：**

```
/etc/shadow       → OwnerUID:0  GroupGID:0  Perms:0600  IsSensitive:true
/home/user1/notes → OwnerUID:1000 GroupGID:1000 Perms:0644  IsSensitive:false
/tmp/data.bin     → OwnerUID:1000 GroupGID:1000 Perms:0666  IsSensitive:false
```

**Unix 权限位检查逻辑（与课本完全一致）：**

```
1. uid == 0            → 通过（root 绕过）
2. uid == file.Owner   → 检查 owner read bit  (Permissions >> 6) & 4
3. gid == file.Group   → 检查 group read bit  (Permissions >> 3) & 4
4. 其他用户             → 检查 other read bit   Permissions & 4
```

校验失败返回 `ValidationError{Code, Description}`，前端在差错控制台实时展示。

## P11 · 故障注入体系

从「正常流转」到「什么情况都可能发生」：

| 故障类型 | 错误码 | 拦截层 | 模拟场景 |
|---------|--------|--------|---------|
| 权限拒绝 | EACCES | VFS 层 | 普通用户读取 `/etc/shadow` |
| 非法地址 | EFAULT | VFS 层 | 用户传入 0xFFFFFFFF 内核空间地址 |
| 硬件超时 | EIO | 硬件层 | 磁道损坏或设备掉线 |
| 路径穿越 | EPERM | VFS 层 | 使用 `../../` 逃逸主目录 |
| 文件不存在 | ENOENT | VFS 层 | 路径在文件系统中不存在 |

运行时注入机制：前端下拉框选择 → 发送 INJECT_FAULT 指令 → 引擎下一步走到对应检查点时触发。

**价值**：EACCES / EFAULT / EIO 这些错误码不再只是课本上的字符串 —— 你可以亲眼看到进程在哪个阶段被拦截、错误如何逐层传递。

[备注：每个故障配一张截图，展示卡片变红 + 差错控制台信息。]

## P15 · 技术难点

**① gRPC-Web WebSocket 帧格式 —— 手写编解码器**

- Improbable 协议帧结构：`[control=0][flag][4B BE len][pb data]`
- 120 行代码替代了 2 万行 npm 依赖，对整个通信路径有完全控制
- 处理了粘包缓冲、消息队列、WebSocket 握手 header 块

**② Canvas 粒子动画坐标系统**

- SVG 是 viewBox 虚拟坐标系，Canvas 是像素坐标系
- 通过 `getBoundingClientRect()` 运行时动态计算所有起终点，不依赖硬编码坐标
- `requestAnimationFrame` 驱动 60fps 贝塞尔曲线粒子运动

**③ CSS 多层弹性布局**

- 右侧 4 个手风琴面板需要在展开时占据剩余垂直空间而不溢出
- 关键技巧：`flex:1` + `height:0` + `min-height:0` 强制从零分配剩余空间

**④ 层级卡片绝对居中**

- 五层卡片宽度各异（330px ~ 560px），需要精确居中于 x=360
- `left:50% + transform:translateX(-50%)` 兼容任意宽度，无需计算偏移量
